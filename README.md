## 它解决什么问题

每逢求职季，真正的时间黑洞不是写简历——而是这四件事同时压过来：

| 问题 | 具体痛点 |
|------|---------|
| **信息散落** | Boss直聘、实习僧分别刷，同一家公司可能在两个平台都出现，手动去重 |
| **筛选低效** | 每天几十条岗位，凭感觉判断"值不值得投"，容易漏掉好机会 |
| **跟进失控** | 投出去的简历不知道在哪个阶段，靠备忘录根本管不住 |
| **打招呼千篇一律** | 复制粘贴，改了公司名忘改岗位名，HR 一眼识破 |

**求职助手**把上述四个环节全部结构化：从"想去哪类岗位"出发，AI 搜索 → AI 评分 → 可视化看板跟进 → 一键投递，用户只做最终判断。

## 核心功能

### 1 · AI 解析求职意向 → 自动生成搜索关键词

用自然语言描述方向（如"数据运营/策略运营/数据分析"），LLM 将其扩展为 6-10 个平台搜索词，涵盖你可能没想到的表达（"运营分析实习""数据产品助理"等），以关键词芯片形式展示、可手动增删。

### 2 · 多平台 × 多城市 × 多关键词全量抓取

- **平台**：Boss直聘、实习僧同时覆盖
- **城市**：一键勾选多个城市，取并集
- **去重**：以 `(平台, 岗位ID)` 为键跨关键词 × 跨城市自动去重
- **活跃度过滤**：可选"仅今日活跃HR"，提升打招呼回复率

### 3 · AI 简历-岗位匹配分析

每条岗位与你的简历全文逐一比对，输出：

| 字段 | 说明 |
|------|------|
| **匹配分（0-100）** | 绿 80+ / 橙 60+ / 红 60- 颜色编码 |
| **匹配亮点** | 2-3 条与岗位高度吻合的简历要素 |
| **潜在顾虑** | 1-2 条可能短板，面试前提前准备 |
| **公司规模** | 大厂 / 中厂 / 小厂，覆盖 100+ 公司关键词 |

### 4 · 看板管理 + 时间线记录

七阶段可视化看板（待投递 → 已投递 → 简历筛选中 → 笔试/测评 → 面试中 → Offer → 未通过），拖拽即更新状态，每个岗位可追加带时间戳的进展备注。

### 5 · AI 打招呼 & 自动投递

- **两种模式**：固定模板（可编辑）或 AI 实时生成个性化文案（60字内，结合岗位要求）
- **发送前预览**：内容可在弹窗内修改，确认后系统自动在 Boss/实习僧完成投递
- **批量投递**：多选岗位一键发出

### 6 · 演示模式

内置 30 条仿真岗位数据（大/中/小厂全覆盖），无需登录任何平台即可体验完整流程。演示数据与真实数据隔离存储，切 Tab 互不干扰。

## 关键技术决策

### 为什么用 Patchright 而不是原生 Playwright？

Boss直聘对 CDP（Chrome DevTools Protocol）有检测，原生 Playwright 会被识别为机器人。Patchright 是 Playwright 的二进制分支，在 Chromium 底层打了反检测补丁，无需注入 `add_init_script`（注入脚本本身也会暴露特征），结合随机 UA、随机视口、人类化鼠标轨迹，显著降低被封风险。

### 为什么 Boss 会话全程复用同一个浏览器 Context？

Boss直聘把登录状态绑定在浏览器指纹上，重新创建 Context 等价于换了一台设备，会立刻掉登录。因此 `_boss_scraper` 在服务启动后全程保持为单例，不关闭、不重建，只在需要时复用同一个 Context 发起新请求。

### 为什么 AI 关键词扩展放在搜索前而不是搜索后？

搜索词的质量决定召回质量。用户输入"数据运营"，如果直接搜这三个字，可能只返回 title 里带"数据运营"的岗位，漏掉"数据策略实习""运营分析助理"等等价表达。LLM 先做语义扩展再交给爬虫，等价于在平台侧做了一次手动关键词矩阵——但不需要用户自己去想。

### 为什么演示数据和真实数据要完全隔离？

混用会导致两个问题：演示数据污染真实统计数字（看板漏斗数不准）；对演示岗位执行"真实投递"时系统找不到合法 URL 而报错。通过 SQLite 的 `is_demo` 列 + 所有查询都带 `WHERE is_demo = ?` 过滤，从存储层隔离，前端两个 Tab 分别加载各自数据，互不干扰。

## 技术栈

| 层 | 工具 | 选择理由 |
|---|---|---|
| 后端框架 | FastAPI | 异步 + BackgroundTasks，搜索/分析任务不阻塞响应 |
| 浏览器自动化 | Patchright（Playwright 分支）| 二进制级反检测，绕过 Boss直聘 CDP 检测 |
| AI 接口 | OpenRouter · DeepSeek V3 | 统一接口可热切换模型，成本低，中文能力强 |
| 数据库 | SQLite + aiosqlite | 零部署，完整异步支持，本地即可运行 |
| 前端 | 原生 JS（无框架）| 无构建工具依赖，加载快，逻辑轻 |

## 快速启动

```bash
# 1. 安装依赖
pip install -r requirements.txt
python -m patchright install chromium

# 2. 配置 API Key
cp .env.example .env
# 编辑 .env，填入 OPENROUTER_API_KEY=sk-or-...

# 3. 启动服务
uvicorn main:app --host 0.0.0.0 --port 8000

# 4. 浏览器打开
open http://localhost:8000
```

**系统要求**：Python 3.11+，macOS / Linux

> 没有 API Key？直接用**演示模式**体验全部功能，无需任何账号或密钥。
